# 前端动态恢复展示改造技术文档

## 1. 文档目的

本文档用于说明当前前端为什么还没有正确表达“换电恢复 / 换机接力”的新后端能力，以及前端接下来应该如何改造。

适用对象：

- 当前 Cesium 前端维护者
- 需要根据后端新字段重写 `CesiumMap.vue` 的开发者

本文档只讨论前端展示逻辑，不修改后端算法。

---

## 2. 当前问题是什么

后端现在已经支持动态恢复调度，能返回：

- 静态执行模式
- 动态恢复模式
- sortie 分段
- 换电/换机事件
- 链级完成状态

但当前前端还是按“旧的静态链级模型”在展示，因此会出现你现在看到的情况：

### 2.1 仍然把整条链当成一次性执行

例如这条链：

- `外卖柜4号 -> 汇元楼 -> 校医院 -> 图书馆南馆 -> 田径运动场 -> 立功门`

后端实际上已经把它拆成了两段甚至多段 sortie，并且：

- 前半段可能由 `drone-1` 执行
- 后半段可能由 `drone-2` 接力执行

但前端仍然沿用旧逻辑：

- 只显示一个统一的 `execution_mode`
- 只显示一套旧链级摘要
- 地图轨迹还是按旧的 chain 顺序去画

所以现在虽然任务列表里已经出现了 `drone-1` / `drone-2` 的变化，但整体展示还是不够准确。

### 2.2 `execution_mode` 已经不够表达真实语义

旧前端默认：

- `direct`
- `after_swap`
- `infeasible`

现在后端新增了更关键的字段：

- `static_execution_mode`
- `dynamic_recovery_mode`
- `final_status`

因此：

- `execution_mode = after_swap`

不再一定表示“执行前换电后整链一次完成”，也可能表示：

- 静态不可执行，但通过动态换电恢复完成
- 静态不可执行，但通过换机接力完成

所以前端不能再只看 `execution_mode`。

### 2.3 节点级累计耗时已经变成“整条链累计完成时间”

当前后端在动态恢复模式下，`cumulative_time_min` 已经按整条 mission 累积。

也就是说：

- `T5`
- `T6`

显示的是恢复之后的整链累计时间，而不是旧版单 sortie 累计。

因此前端也要同步切换到“mission 视角”，不能继续按“单次起飞视角”理解。

---

## 3. 后端现在提供了哪些新字段

当前后端响应中，任务级和链级主要新增了这些字段。

## 3.1 任务级字段

每个任务节点现在可能包含：

- `static_execution_mode`
- `dynamic_recovery_mode`
- `final_status`
- `sortie_id`
- `recovery_action_before_task`
- `chain_total_completion_time_min`
- `swap_count`
- `handoff_count`
- `recovery_event_summary`

字段含义如下：

### `static_execution_mode`

表示旧静态链级判断结果：

- `direct`
- `after_swap`
- `infeasible`

### `dynamic_recovery_mode`

表示动态恢复策略：

- `none`
- `swap_recovery`
- `handoff_recovery`
- `mixed_recovery`

### `final_status`

表示最终链完成结果：

- `completed_directly`
- `completed_after_swap_recovery`
- `completed_after_handoff_recovery`
- `completed_after_mixed_recovery`
- `failed_even_with_recovery`

### `sortie_id`

表示该任务属于第几次 sortie。

例如：

- `T1 ~ T4` 属于 `sortie 1`
- `T5 ~ T6` 属于 `sortie 2`

### `recovery_action_before_task`

表示该任务之前是否刚发生恢复动作：

- `none`
- `swap`
- `handoff`

### `recovery_event_summary`

这是给前端直接展示的人类可读描述，例如：

- `返航中心站换电 2.00 min`
- `返回中心站后由 drone-2 接替执行，交接 1.00 min`

---

## 3.2 链级字段

响应顶层还新增了：

- `chain_summaries`

每条链都会有一条 `ChainExecutionSummary`。

里面主要包含：

- `chain_root_task_id`
- `static_execution_mode`
- `dynamic_recovery_mode`
- `final_status`
- `total_completion_time_min`
- `final_return_margin_min`
- `swap_count`
- `handoff_count`
- `used_drone_ids`
- `sorties`
- `recovery_events`

这部分是前端重构的关键数据来源。

---

## 4. 前端为什么必须改

结论很明确：

> 是的，前端必须改。

因为后端已经从“静态链级判定器”升级成“动态恢复调度器”了，而前端还停留在旧版本思路。

如果不改，就会出现：

- 轨迹显示与真实调度不一致
- 标签语义不准确
- 链级摘要误导用户
- 无法清楚展示换电/换机事件

---

## 5. 前端改造目标

改造后的前端需要实现四件事：

### 5.1 从“单链一次执行”改成“多 sortie 展示”

也就是说：

- 一条链不再只画一条连续路径
- 要按 `sortie_id` 分段绘制

### 5.2 显示恢复事件

要能清楚展示：

- 在中心站换电
- 在中心站换机接力

### 5.3 区分“静态结果”和“最终结果”

任务卡片和链级摘要都要同时表达：

- 静态上本来能不能飞
- 动态恢复后最终是否完成

### 5.4 地图上准确显示无人机切换

如果后半段由另一架无人机执行：

- 路径颜色要变
- 标签也要变
- 不能再把整条链当成同一架机执行

---

## 6. 推荐的前端展示方案

## 6.1 任务列表展示方案

每条任务卡片建议展示为三层：

### 第一层：任务基础信息

继续保留：

- `task_id`
- `location_name`
- `assigned_drone_id`
- `sortie_id`

例如：

- `T5`
- `田径运动场`
- `drone-2`
- `Sortie 2`

### 第二层：链级恢复摘要

在链头任务上显示：

- 静态执行模式：`static_execution_mode`
- 动态恢复模式：`dynamic_recovery_mode`
- 最终状态：`final_status`
- 总完成时间：`chain_total_completion_time_min`
- 总换电次数：`swap_count`
- 总换机次数：`handoff_count`

例如：

- 静态结果：`infeasible`
- 恢复方式：`swap_recovery`
- 最终结果：`completed_after_swap_recovery`
- 总完成时间：`42.20 min`
- 换电次数：`1`

### 第三层：节点级信息

继续显示你现在已有的：

- 当前段
- 本段距离 / 耗时
- 当前节点执行耗时
- 累计耗时
- 执行后剩余电量
- 当前点返航需求

如果该任务前刚好发生恢复事件，还要额外显示：

- `recovery_event_summary`

例如：

- `返航中心站换电 2.00 min`
- `返回中心站后由 drone-2 接替执行，交接 1.00 min`

---

## 6.2 地图展示方案

地图要改成按 sortie 绘制。

### 旧逻辑

旧逻辑是：

- 从中心站开始
- 沿着整条 chain 顺序画到底
- 最后一次返航

### 新逻辑

新逻辑要变成：

- 根据 `chain_summaries[].sorties` 分段绘制
- 每个 sortie：
  - 从中心站出发
  - 依次飞到该 sortie 内的任务节点
  - 再返航中心站

也就是说：

如果一条链被拆成两段：

- `sortie 1`
- `sortie 2`

地图上就要画成两次独立出发和返航。

---

## 6.3 颜色策略

仍然建议保持：

- 颜色按 `assigned_drone_id`

这样一旦发生换机：

- 前半段 `drone-1`
- 后半段 `drone-2`

用户会一眼看出来。

### 线型策略建议

可以这样设计：

- 正常 sortie 主路径：实线
- 返航中心站：虚线
- 恢复事件本身不画飞行线，只在中心站用标签表示

### 恢复事件标记

在中心站增加标签：

- `换电 2 min`
- `换机 1 min`

如果一个链中发生多个恢复事件，可以按时间顺序叠加。

---

## 7. 前端具体应该怎么改

这里以 `CesiumMap.vue` 为主要改造对象。

## 7.1 数据处理层

当前前端拿到 `taskList` 后，主要还是按旧逻辑自己推 chain。

建议改成：

### 优先使用后端返回的 `chain_summaries`

理由：

- 后端已经给出了真正的 sortie 划分
- 前端不应该再自己猜

前端应新增：

- `const chainSummaries = ref([])`

接口返回后：

```js
taskList.value = res.data.tasks || []
chainSummaries.value = res.data.chain_summaries || []
```

---

## 7.2 任务列表层

修改任务列表模板：

### 链头任务显示恢复摘要

条件：

- `t.show_chain_summary === true`

新增显示：

- `t.static_execution_mode`
- `t.dynamic_recovery_mode`
- `t.final_status`
- `t.chain_total_completion_time_min`
- `t.swap_count`
- `t.handoff_count`

### 普通任务显示 sortie 信息

新增显示：

- `t.sortie_id`
- `t.recovery_event_summary`（如果有）

---

## 7.3 地图仿真层

当前的 `runSimulation(tasks)` 必须改。

### 旧逻辑问题

它现在是：

- 自己根据 `predecessor_id` 组链
- 然后把链一次画完

这已经不够用了。

### 新逻辑建议

新增一个方法，例如：

```js
const runRecoverySimulation = (tasks, chainSummaries) => { ... }
```

核心流程：

1. 清空 viewer
2. 先画中心站
3. 为每条 `chainSummary` 遍历 `sorties`
4. 每个 sortie：
   - 读取 `assigned_drone_id`
   - 读取该 sortie 的 `task_ids`
   - 找到这些任务的坐标
   - 从中心站出发逐段画线
   - sortie 最后返航中心站
5. 如有 `recovery_events`：
   - 在中心站打标签或临时 marker

---

## 7.4 sortie 绘制逻辑

对每个 `SortieSegment`：

### 起点

- 固定是 `HOME_BASE`

### 中间点

- `sortie.task_ids` 对应的任务节点坐标

### 终点

- 完成最后一个任务后回 `HOME_BASE`

### 标签

建议给每个 sortie 增加：

- `drone-1 / sortie 1`
- `drone-2 / sortie 2`

---

## 8. 前端推荐新增的格式化函数

建议新增几个展示辅助函数：

### 8.1 静态模式标签

```js
const getStaticModeLabel = (mode) => { ... }
```

映射：

- `direct` -> `静态可直接执行`
- `after_swap` -> `静态需执行前换电`
- `infeasible` -> `静态不可执行`

### 8.2 动态恢复标签

```js
const getDynamicRecoveryLabel = (mode) => { ... }
```

映射：

- `none` -> `无需动态恢复`
- `swap_recovery` -> `换电恢复`
- `handoff_recovery` -> `换机接力`
- `mixed_recovery` -> `混合恢复`

### 8.3 最终状态标签

```js
const getFinalStatusLabel = (status) => { ... }
```

映射：

- `completed_directly`
- `completed_after_swap_recovery`
- `completed_after_handoff_recovery`
- `completed_after_mixed_recovery`
- `failed_even_with_recovery`

## 9. 前端改造后的理想效果

以你现在这条高压链为例，前端最终应该能表达成：

### 链头摘要

- 静态结果：`infeasible`
- 动态恢复：`handoff_recovery` 或 `swap_recovery`
- 最终状态：`completed_after_*`
- 总完成时间：`xx.xx min`
- 换电次数：`1`
- 换机次数：`1`

### 任务列表

- `T1 ~ T4` 属于 `Sortie 1 / drone-1`
- `T5 ~ T6` 属于 `Sortie 2 / drone-2`
- `T5` 前显示：
  - `返回中心站后由 drone-2 接替执行，交接 1.00 min`

### 地图

- 第一段路线是 `drone-1` 的颜色
- 第二段路线是 `drone-2` 的颜色
- 中心站处有恢复事件标记

---

## 10. 这次前端改造的边界

本次前端改造建议只做以下内容：

- 正确展示动态恢复字段
- 按 sortie 分段绘制路径
- 正确显示换电 / 换机事件

暂时不建议一起做：

- 时间轴事件动画
- 中心站事件弹窗
- 多链恢复优先级排序可视化
- 甘特图或任务时间轴

这些可以放后续增强版。

---

## 11. 推荐的实现顺序

### 第一步

先修改接口接收逻辑，保存：

- `tasks`
- `chain_summaries`

### 第二步

重写任务列表摘要显示：

- 加入静态结果、动态恢复模式、最终状态

### 第三步

重写地图绘制逻辑：

- 从按 chain 画
- 改成按 sortie 画

### 第四步

补中心站恢复事件标记：

- `换电 2 min`
- `换机 1 min`

### 第五步

最后再优化样式和文案。

---

## 12. 一句话总结

当前前端确实还没有完成和新后端的对齐。

最核心的问题是：

> 后端已经进入“动态恢复调度”阶段，而前端还停留在“静态链级执行”展示阶段。

因此前端下一步必须改成：

- 使用 `chain_summaries`
- 按 `sortie` 分段绘制
- 显示 `swap / handoff` 恢复事件
- 同时区分静态结果与最终完成结果

只有这样，新的换电恢复和换机接力能力才能被完整、准确地展示出来。
