# 无人机换电恢复技术文档

## 1. 文档目的

本文档用于说明“无人机中途返航换电后继续执行任务链”这一功能的设计目标、算法思路和代码实现方案。

它面向当前项目的后续开发，核心目标是把现有的“静态链级电量判定”升级为“可恢复的动态执行调度”。

---

## 2. 该功能要解决什么问题

当前系统已经支持三种静态判定结果：

- `direct`：当前电量足以直接完成整条任务链
- `after_swap`：当前电量不足，但在执行前先换电后可以完成整条任务链
- `infeasible`：即使满电也无法一次性完成整条任务链

但这套逻辑仍然是“一次 sortie 执行完整条链”的假设。

这会带来一个局限：

- 某些任务链虽然无法一次满电完成
- 但如果允许无人机执行到一部分后返航换电
- 再从中心站继续执行剩余任务
- 那么整条任务链实际上是可完成的

因此，换电恢复功能的意义是：

> 将原本静态判定为不可一次完成的长链任务，改造成可分阶段完成的动态任务链。

---

## 3. 功能定义

### 3.1 核心设定

- 所有无人机都从“无人机中心站点”起飞
- 当当前 sortie 无法继续执行后续任务时，无人机先返回中心站点
- 在中心站完成换电
- 换电耗时固定设为 `2 min`
- 换电后同一架无人机继续执行剩余任务链

### 3.2 功能边界

当前版本建议只支持：

- 返航到中心站后换电
- 同一架无人机继续执行剩余任务链
- 每次恢复动作都发生在中心站

当前版本不建议支持：

- 空中换电
- 中途任意地点换电
- 不回中心站的临时恢复
- 同时引入换电和换机的复杂联合优化

---

## 4. 功能可以实现什么效果

引入换电恢复后，系统将从“单次链级可行性判断”升级为“多 sortie 动态恢复执行”。

具体可实现：

### 4.1 将部分静态 `infeasible` 任务恢复为可完成任务

例如一条链：

`中心站 -> T1 -> T2 -> T3 -> T4 -> T5 -> 中心站`

若满电只能完成到 `T3`，则可变为：

- 第 1 次 sortie：`中心站 -> T1 -> T2 -> T3 -> 中心站`
- 中心站换电 `2 min`
- 第 2 次 sortie：`中心站 -> T4 -> T5 -> 中心站`

### 4.2 给出更真实的调度结果

系统不再简单返回：

- `能飞`
- `不能飞`

而是可以返回：

- 一次性直接完成
- 需要中途返航换电后继续完成
- 即使分段恢复也无法完成

### 4.3 支持更真实的前端可视化

前端可以表达：

- 第几次 sortie
- 在中心站发生换电
- 换电耗时 2 分钟
- 恢复后继续执行的后续路径

---

## 5. 算法思路

## 5.1 总体思想

将一条任务链视为一个有序任务序列：

`T1 -> T2 -> T3 -> ... -> Tn`

系统不再要求一架无人机一次性完成全部节点，而是允许：

- 每次 sortie 完成一段“最长可执行前缀”
- 然后返航中心站
- 再通过换电继续执行后续未完成部分

### 5.2 核心算法目标

给定：

- 当前无人机状态
- 中心站坐标
- 任务链节点序列
- 换电耗时 `2 min`
- 安全返航余量

输出：

- 该任务链是否可通过多次换电恢复完成
- 需要几次 sortie
- 每次 sortie 执行哪些任务
- 总完成时间
- 总换电次数

---

## 6. 算法流程

### Step 1：初始化

输入：

- 一条任务链 `chain`
- 一架当前执行无人机 `drone`
- 中心站坐标 `home_base`

初始化变量：

- `remaining_tasks = chain`
- `sortie_id = 1`
- `swap_count = 0`
- `total_completion_time = 0`

### Step 2：寻找当前 sortie 的最长可执行前缀

从 `remaining_tasks` 的第一个任务开始，逐个尝试加入当前 sortie。

对于每个候选前缀，都要计算：

- 从当前起点到该前缀最后一个任务的飞行时间
- 前缀内部所有任务的执行时间
- 从前缀最后一个任务返航中心站的时间
- 安全返航余量

若满足：

`当前电量 >= 前缀总时间`

则继续尝试加入下一个任务。

若不满足，则停止。

最终得到：

- 当前 sortie 能完成的最长前缀

### Step 3：如果一个任务都完成不了，直接失败

如果当前电量连：

- 从中心站出发
- 完成下一个任务
- 再安全返航中心站

都做不到，则该链动态恢复失败。

此时可返回：

- `final_status = infeasible`
- `reason = current battery cannot safely complete even one recoverable segment`

### Step 4：执行该 sortie 并返航

将该前缀从 `remaining_tasks` 中移除。

记录：

- 本次 sortie 起点
- 本次 sortie 终点
- 完成的任务节点列表
- 返回中心站时间

将这部分时间累加到：

- `total_completion_time`

### Step 5：判断是否已完成整条任务链

如果 `remaining_tasks` 为空：

- 任务链完成
- 返回 `recoverable_success`

否则进入下一步。

### Step 6：执行换电恢复

在中心站进行换电：

- 固定耗时 `2 min`
- 无人机电量恢复为满电 `30 min`

更新：

- `swap_count += 1`
- `total_completion_time += 2`
- `sortie_id += 1`

然后继续 Step 2，处理剩余任务链。

---

## 7. 结果判定方式

引入换电恢复后，建议把原先的执行状态扩展为两层结构。

### 7.1 保留原静态判定

用于和现有系统兼容：

- `direct`
- `after_swap`
- `infeasible`

### 7.2 新增动态任务完成状态

建议新增链级字段：

- `static_execution_mode`
- `dynamic_recovery_mode`
- `final_status`

其中：

- `static_execution_mode`
  - 原静态结果
- `dynamic_recovery_mode`
  - `none`
  - `swap_recovery`
- `final_status`
  - `completed_directly`
  - `completed_after_swap_recovery`
  - `failed_even_with_swap_recovery`

## 8. 数据结构建议

建议新增一组链级结构，不必强行塞进单个任务字段。

### 8.1 建议新增的 sortie 结构

```python
class SortieSegment(BaseModel):

    assigned_drone_id: str
    start_from_home: bool

    segment_end_name: str

    return_to_home_time_min: float
```

```python
class RecoveryEvent(BaseModel):
    event_type: Literal["swap"]

    duration_min: float
    after_sortie_id: int
```

## 9. 代码实现建议

## 9.1 推荐新增模块

建议新增一个独立模块，例如：

- `src/resource_recovery_scheduler.py`

不要直接把全部动态调度逻辑堆进当前的 [resource_evaluator.py](d:\SZU-project\UAV_Agent\src\resource_evaluator.py)。

### 该模块职责

- 输入：任务链、无人机状态、中心站参数
- 输出：动态恢复调度结果

---

## 9.2 推荐复用现有函数

当前系统已有：

- `_build_task_chains()`
- `_estimate_chain_total_time()`
- `estimate_exec_time_min()`
- `calculate_distance_m()`

建议复用方式：

- 静态判定继续保留在 `resource_evaluator.py`
- 动态恢复算法在新模块中调用现有时间/距离计算函数

---

建议新增：

```python
def evaluate_with_swap_recovery(
    tasks: Iterable[TaskTuple],
    drones: list[DroneState],
    full_battery_min: float = FULL_BATTERY_MIN,
    swap_duration_min: float = 2.0,

    ...
```

功能：

- 先做静态链级判定
- 对静态 `direct` 和 `after_swap` 保持原逻辑
- 对静态 `infeasible` 的长链，进一步尝试动态换电恢复

## 11. 实验如何做

新增这类实验：

### 11.1 Static infeasible -> Dynamic recoverable

设计一批样本：

- 静态一次性飞不完
- 但通过 2 次 sortie 可以完成

指标：

- `Recovery Success Rate`
- `Average Swap Count`
- `Average Completion Time`

### 11.2 与静态电量模块对比

对比组：

- `Static battery-aware`
- `Dynamic swap recovery`

证明：

- 静态不可执行任务中，有一部分可以通过换电恢复完成

---

## 12. 论文中的表述建议

建议把这个功能写成：

> 基于中心站恢复的动态换电调度机制

不要写成：

> 任意中途换电

因为你当前假设很明确：

- 所有恢复动作都在中心站完成

论文中可表述为：

> 当任务链无法在单次 sortie 内完成时，系统允许无人机返回中心站进行换电，并以多 sortie 方式继续执行剩余任务，从而提升长链任务的可完成性。

---

## 13. 一句话总结

换电恢复功能的本质是：

> 将原本“一次性执行整条任务链”的静态判定，升级为“允许返航换电后继续执行剩余任务”的动态恢复调度。

它是当前电池模块从“可行性评估器”走向“恢复型调度器”的第一步。
