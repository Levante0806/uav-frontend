# Spatial GraphRAG 后端技术实施与静态数据规格说明书

## 1. 工作背景与核心目标

随着无人机项目调度范围从单一微观场景（深圳大学校园）扩展至宏观的城市区域（如粤海街道），原有的前端硬编码与纯人工获取坐标点的方式已无法满足需求。为了实现**城市级无人机的自动化调配、合法路径规划及防碰撞避障**，系统决定引入 **Spatial GraphRAG (空间图谱检索增强)** 架构。

该架构将以真实的城市地理数据（建筑物轮廓、路网骨架）作为基础，构建起包含物理空间拓扑属性的“无人机空间知识图谱”。在此基础上，结合大语言模型（LLM）的检索与推理能力，使得无人机系统具备理解自然语言指令、自动感知空间障碍物、并执行坐标纠偏的能力，具备极高的学术科研与工程创新价值。

## 2. 整体职责划分

为保证生态契合度和系统运行效率，GraphRAG 的核心运算应完全由 **后端（或独立 Agent 模块）** 承接。

1. **后端 / Agent 层（Python 生态）**：作为无人机系统的“大脑”。负责解析原始 GIS 地图数据，利用 `geopandas` 和 `networkx` 在内存或图数据库（如 Neo4j）中构建完整的包含建筑物、街道的拓扑网络。后续阶段还将负责：文本向量化 embedding、结合语言模型的空间关系推理过滤，以及最终向前端暴露安全的 JSON 坐标系列或路径数据。
2. **前端层（Vue + Cesium）**：作为无人机的“眼睛”。仅作为纯粹的数据消费与渲染终端。收到后端计算好的合法坐标和路径后，执行三维漫游与 UI 交互，严禁在前端进行重度空间计算与排查操作。

## 3. 详细数据需求与规格规范

考虑到调用开源地图数据不够精确的劣势，我们计划**直接采用研究院现有的、高精度的城市空间底图数据（粤海街道）**来进行二次融合建模。对于底层数据的提供格式与字段，制定如下标准：

### 3.1 数据格式要求
推荐提供业界标准的 GIS 矢量格式：
*   **首选**：`.shp` (Esri Shapefile，提供配套的 .dbf, .shx 等完整同名文件集合)
*   **次选**：`.geojson` 或 `.json` (标准的 GeoJSON 格式)

### 3.2 坐标系强制规范 (CRS)
由于前端渲染引擎 Cesium 以及无人机飞行姿态飞控均采用纯净的地理经纬度，**数据必须带有明确的坐标系说明**。
*   **目标坐标系**：**WGS84** (EPSG:4326)
*   > [!WARNING]
    > **偏差警告：** 若研究院提供的是商用地图常用的 **GCJ-02 (高德/腾讯 火星坐标系)** 或 **CGCS2000**，必须在交付数据时明确标注。后端在清洗数据入库前必须编写转换代码将其还原为 WGS84，否则无人机轨迹在图谱上将出现巨大的建筑物穿模偏移！

### 3.3 具体图层需求分解 (Layers & Features)

#### A. 核心拦截层：建筑物多边形 (Building Footprints)
用于构建无人机飞行不可逾越的“实体障碍边界”，以及检索符合条件的特定作业起降平台。
*   **几何表现**：多边形 (Polygon) 或多多边形 (MultiPolygon)
*   **核心属性需求**：
    *   `Id/Name`：实体编号或具体大厦名称（如“腾讯海滨大厦”、“深大南区宿舍”）。
    *   `Height/Levels` (极度重要)：楼层高度或相对地面高度。以便三维空间规划排查。
    *   `Type`：建筑类型（住宅/商用/政府机构）。

#### B. 骨架拓扑层：城市道路网络 (Road Networks)
出于地面安全等考量，大多数无人机路径需依赖城市骨干马路上空作为“飞行虚拟走廊”。
*   **几何表现**：线串 (LineString) 或多线串 (MultiLineString)
*   **核心属性需求**：
    *   `Road_Name`：道路名称（如“科苑南路”）。
    *   `Road_Class`：道路等级（主干路、次干路、居民区小道）。

#### C. 服务兴趣点层：特定 POI (Points of Interest) - *(选需)*
作为任务链执行的关键停靠点补充。
*   **几何表现**：散点 (Point)
*   **场景需求**：无人机固定起降台位置、已备案的充电补给基站位置、或者一些核心广场的中心集结点。

## 4. 后续工作进展规划阶段 (Roadmap)

在明确数据准备内容后，后端的整体工程实施步骤分为四个阶段逐步拉齐：

### Phase 1: 数据加载与预处理清洗
1.  **动作**：后端人员接收研究院提供的 `.shp` 格式粤海街道数据。
2.  **实现**：使用 Python 编写脚本，借助 `geopandas` 库读取多边形和路网数据。通过数据清洗消除不合法几何体，并将所有发现非 WGS84 坐标系的数据强转为 WGS84 格式输出。

### Phase 2: 组装基本空间知识图谱
1.  **动作**：在内存中建立一张有空间连接特性的 Graph。将路网节点设为 `RoadNode`，将其用线连接为 `Edge`（对应道路）；计算所有建筑物中心坐标，将其作为 `BuildingNode` 抛入图中。
2.  **空间计算实现**：使用 `shapely` 与距离算法，为每栋建筑（Building）找到距离其最近的一个路口节点（RoadNode），为其两端连接一条代表空间连通的 `<Access_Edge>`。最终将此完整网络导出静态 JSON 文件供快速加载。

### Phase 3: 构建基于大模型的 Spatial RAG 引擎
1.  **节点 Embedding**：引入 `LangChain` 或 `LlamaIndex`，对上述 Graph 中的各节点属性（“科苑地铁站旁边限高100米的写字楼”）进行向量化并存入向量库（如 Milvus / Chroma）。
2.  **大语言模型融合**：当前端发送调度控制指令或查询需求时（“找一个离高新园最近的降落点，不要穿过商场”），由 LLM 负责语义解析意图，随后调度空间库对特定范围进行查询约束，返回满足该业务语境的确切合法坐标。

### Phase 4: API 接口与前端对接验证
1.  **端到端连通**：提供 `/api/v1/uav/route_planning` 类似 API，前端发送区域查询请求并接收 Graph Nodes 并在 Cesium 画布中落点渲染。
2.  **强表现力纠偏 Demo**：人为构造偏移坐标事件，故意让无人机撞向楼宇区域，后端 Spatial RAG 图谱触发警告并自动将坐标点平移至合法飞行廊道内并同步给前端，展现技术深层次价值，作为核心论文点输出。

> [!TIP]
> **致后端/Agent 开发组建议**：初期验证只需在单机环境利用 `NetworkX` 挂载本地数据图即可跑通 Phase 2 闭环，Neo4j 等重量级组件可待系统复杂后再行迁移引入。
